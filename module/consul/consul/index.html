简介

该项目通过自动配置和绑定到Spring Environment和其他Spring编程模型习惯用法，为Spring Boot应用程序提供Consul集成。通过一些简单的注释，您可以快速启用和配置应用程序中的通用模式，并使用基于Consul的组件构建大型分布式系统。提供的模式包括服务发现、控制总线和配置。智能路由和客户端负载均衡，断路器是与其他Spring Cloud项目集成提供的。

服务发现


如果您使用Spring Cloud Consul Config，并且您已经设置了 `spring.cloud.bootstrap.enabled=true` 或 `spring.config.use-legacy-processing=true`。或者使用`spring-cloud-starter-bootstrap`，那么上面的值将需要放在`bootstrap.yml`中而不是`application.yml`。

注册

当客户机向Consul注册时，它会提供关于自身的元数据，如主机和端口、id、名称和标记。默认情况下创建一个HTTP Check，领事每10秒访问一次/执行器/运行状况端点。如果健康检查失败，服务实例将被标记为critical。

如果Consul客户端位于localhost:8500之外的其他地方，则需要配置来定位客户端。

application.yml

spring:
  cloud:
    consul:
      host: localhost
      port: 8500

  • ${Spring .application.name} 服务名
  • Spring Context ID 实例 ID
  • ${server.port} 端口
  • spring.cloud.consul.discovery.enabled 禁用Consul发现客户端
  • spring.cloud.consul.discovery.register 禁用服务注册

将管理注册为单独的服务

管理服务将从应用服务继承其instanceId和serviceName

spring:
  application:
    name: myApp
management:
  server:
    port: 4452
spring:
  cloud:
    consul:
      discovery:
        instance-id: custom-service-id
        serviceName: myprefix-${spring.application.name}

以上配置将注册以下2个服务:

  • Application Service:
ID: custom-service-id
Name: myprefix-myApp
  • Management Service:
ID: custom-service-id-management
Name: myprefix-myApp-management

  • spring.cloud.consul.discovery.management-port 注册管理服务的端口(默认为 management 端口)
  • spring.cloud.consul.discovery.management-suffix 注册管理服务时使用的后缀(默认为“management”)
  • spring.cloud.consul.discovery.management-tags 在注册管理服务时使用的标签(默认为“management”)

HTTP健康检查

Consul实例的健康状况检查默认为“/actuator/health”

spring:
  cloud:
    consul:
      discovery:
        healthCheckPath: ${management.server.servlet.context-path}/actuator/health
        healthCheckInterval: 15s
        healthCheckHeaders:
            X-Config-Token: 6442e58b-d1ea-182e-cfa5-cf9cddef0722

  • spring.cloud.consul.discovery.healthCheckPath HTTP 健康检查状况路径
  • spring.cloud.consul.discovery.healthCheckInterval HTTP 健康检查时间间隔
  • spring.cloud.consul.discovery.healthCheckHeaders HTTP 头文件(可为数组)
  • spring.cloud.consul.discovery.health-check-* 其他设置
  • spring.cloud.consul.discovery.register-health-check 禁用HTTP健康检查

致动器健康指示器

如果服务实例是Spring Boot致动器应用程序，则可能会向其提供以下致动器运行状况指示器。

DiscoveryClientHealthIndicator

当领事服务发现处于活动状态时，将配置DiscoverClientHealthIndicator，并使其可用于执行器运行状况端点。请参阅这里的配置选项。

ConsulHealthIndicator

配置了一个指示器，用于验证ConsulClient的运行状况。

  • spring.cloud.consul.health.indicator.include-services-query 跳过服务检索，只检查领导节点状态(默认情况下，它检索Consul领导节点状态和所有注册服务。在拥有许多注册服务的部署中，在每次运行状况检查中检索所有服务的成本可能很高。)
  • management.health.consul.enabled=false 禁用指标

Metadata

Spring Cloud的ServiceInstance有一个Map<String, String>由服务元字段填充的元数据字段。

spring:
  cloud:
    consul:
      discovery:
        metadata:
          key: value

key -> value

实例ID 唯一

  • spring.cloud.consul.discovery.instanceid 设置实例ID唯一标识符(默认情况下，Spring Application Context ID是${spring.application.name}:comma，separated,profiles:${server.port})

服务发现客户端 DiscoveryClient

org.springframework.cloud.client.discovery.DiscoveryClient，它为发现客户端提供了一个简单的API

@Autowired
private DiscoveryClient discoveryClient;

public String serviceUrl() {
    List<ServiceInstance> list = discoveryClient.getInstances("STORES");
    if (list != null && list.size() > 0 ) {
        return list.get(0).getUri();
    }
    return null;
}

服务监视

Consul Catalog Watch利用了Consul监视服务的能力。Catalog Watch执行一个阻塞的Consul HTTP API调用，以确定是否有任何服务发生了更改。如果有新的业务数据，则发布Heartbeat Event。

  • change spring.cloud.consul.config.discovery.catalog-services-watch-delay 监视频率频率。缺省值是1000，单位是毫秒。延迟是前一个调用结束和下一个调用开始之后的时间量
  • spring.cloud.consul.discovery.catalogServicesWatch.enabled=false 禁用目录监视
  • Watch 使用一个Spring TaskScheduler来安排对consul的调用。默认情况下，它是一个poolSize为1的ThreadPoolTaskScheduler。要更改TaskScheduler，创建一个名为ConsulDiscoveryClientConfiguration.CATALOG_WATCH_TASK_SCHEDULER_NAME的TaskScheduler类型的bean。常数。

配置中心

Consul为存储配置和其他元数据提供了一个键/值存储。Spring Cloud Consul Config是配置服务器和客户端的替代方案。

配置在特殊的“引导”阶段加载到Spring环境中。默认情况下，配置文件保存在/config文件夹中。根据应用程序的名称和模仿Spring Cloud Config解析属性顺序的活动概要文件创建多个PropertySource实例。

例如，一个名为“testApp”，配置文件名为“dev”的应用程序将创建以下属性源:

config/testApp,dev/
config/testApp/
config/application,dev/
config/application/

最特定的属性源位于顶部，最不特定的属性源位于底部 。config/application文件夹中的属性适用于使用consul进行配置的所有应用程序 。config/testApp文件夹中的属性只对名为“testApp”的服务实例可用 。

配置数据导入

spring.config.import属性导入配置数据的新方法。

application.properties

spring.config.import=optional:consul:

默认导入本地即 http://localhost:8500, 可修改

spring.config.import=optional:consul:myhost:8500/context/one;/context/two

这将选择只从/context/one和/context/two加载配置。

个性化

Consul配置可以使用以下属性自定义:

spring:
  cloud:
    consul:
      config:
        enabled: true
        prefix: configuration
        defaultContext: apps
        profileSeparator: '::'

  • enabled将此值设置为“false”将禁用Consul配置
  • prefix为配置值设置基本文件夹
  • defaultContext设置所有应用程序使用的文件夹名称
  • profilesseparator设置分隔符的值，该分隔符用于将属性源中的配置文件名称与配置文件分开

配置监视

Consul Config Watch利用Consul监视键前缀 的能力。

  • spring.cloud.consul.config.watch.delay 监视频率 缺省值是1000，单位是毫秒。延迟是前一个调用结束和下一个调用开始之后的时间量。
  • spring.cloud.consul.config.watch.enabled=false 禁用配置监视设置
  • Watch 使用一个Spring TaskScheduler来安排对领事的调用。默认情况下，它是一个poolSize为1的ThreadPoolTaskScheduler。要更改TaskScheduler，创建一个名为常数 ConsulConfigAutoConfiguration.CONFIG_WATCH_TASK_SCHEDULER_NAME类型为TaskScheduler的bean。

YAML 或 Properties 配置

相对于单独的键/值对，用YAML或properties格式存储一组属性可能更方便。

将spring.cloud.conf.config.format属性设置为YAML或PROPERTIES。

spring:
  cloud:
    consul:
      config:
        format: YAML

必须在consul中的适当data键中设置YAML。使用上面的默认键看起来像:

config/testApp,dev/data
config/testApp/data
config/application,dev/data
config/application/data

您可以使用spring.cloud.consul.config.data-key更改数据键。

git2consul 配置

git2consul是一个Consul社区项目，它将文件从一个git存储库加载到Consul的各个密钥中。缺省情况下，密钥的名称是文件的名称。支持YAML和Properties文件，文件扩展名分别为.yml和. properties。将spring.cloud.conf.config.format属性设置为FILES。

spring:
  cloud:
    consul:
      config:
        format: FILES

在/config中给定以下关键字，development概要文件和一个foo的应用程序名称:

.gitignore
application.yml
bar.properties
foo-development.properties
foo-production.yml
foo.properties
master.ref

将创建以下属性源:

config/foo-development.properties
config/foo.properties
config/application.yml

每个键的值需要是一个正确格式化的YAML或Properties文件。

重试

如果您期望 Consul Agent 在您的应用程序启动时偶尔不可用，您可以要求它在失败后继续尝试。您需要将spring-retry和spring-boot-starter-aop添加到类路径中。默认行为是重试6次，初始回退间隔为1000ms，后续回退的指数乘数为1.1。您可以使用spring.cloud.consul.retry.*
